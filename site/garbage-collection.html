A language like Lox is “high-level” because it frees programmers from having to worry about details that are too mundane or irrelevant to what they are trying to accomplish. The user becomes an executive giving the machine abstract goals and letting the lowly computer figure how to get there.

Dynamic memory allocation is a perfect candidate for automation. It’s necessary for a working program, typically tedious to do by hand, and yet still error-prone. When errors do happen, it can be catastrophic, leading to crashes, memory corruption or security violations. It’s the kind of risky-yet-boring work that machines excel at over humans.

This is why Lox is a “managed” language, which means that the language implementation manages memory allocation on the user’s behalf. The way this works is that any time a user performs an operation that requires some dynamic memory, the allocation is done automatically and implicitly. Whenever they need some memory, it’s handed out. They never worry about deallocating anything. As long as they are using a piece of memory, the machine ensures that memory is still there.

Lox provides the illusion that the computer has an infinite amount of memory. Users can allocate and allocate and allocate and never once think about where all these bytes are coming from. Of course, computers do not yet have infinite memory. So the way managed languages maintain this illusion is by going behind the programmer’s back and reclaiming memory that the program no longer needs. The component that does this is called a garbage collector.

Recycling would really be a better metaphor for this. The GC doesn’t throw away the memory, it frees it up so it can be reused for new data in the future. But managed languages have been around since before Earth Day was started, so they went with the similies they had at hand.

26 . 0 . 1 Reachability

This raises a surprisingly difficult question: how does a VM tell what memory isn’t needed? Memory can only be needed if it will be read from in the future, but short of having a time machine, how can an implementation tell what code the program will execute and which data it will read? Spoiler alert: VMs cannot travel into the future. Instead, the language makes a conservative approximation: it considers a piece of memory to still be in use if it could possibly be read in the future.

I’m using “conservative” in the general sense. There is such a thing as “conservative garbage collector” which means something more specific that what I’m talking about here. All garbage collectors are “conservative” in that they keep memory alive if it could be accessed instead of having a Magic 8-Ball that lets them more precisely know what data will be accessed.

A conservative GC is a special kind of collector that considers any piece of memory to be a pointer if the value in there looks like it could be an address. This is in contrast to a precise GC—which is what we’ll implement—that knows exactly which words in memory are pointers and which store other kinds of values like numbers or strings.

That seems really conservative. Couldn’t any bit of memory possibly be read? Actually, no. Here’s an example:

var a = "first value";
a = "updated";
print a;

Pause this program right after the assignment has completed on the second line. We want to run the GC to reclaim any memory we can right then. The string “first value” is still sitting in memory, but there is no way for the user’s program to ever get to it. Once a got reassigned, the program lost its reference to that string. It is unreachable and we can safely free it. A value is reachable if there is some way for a user program to reference it.

Many values can be directly accessed by the VM. Take a look at:

var global = "string";
{
  var local = "another";
  print global + local;
}

Pause the program right after the two strings have been concatenated but before the print statement has executed. The VM can reach "string" by looking through the global variable table and finding the entry for global. It can find "another" by walking its value stack and hitting the slot for the local variable local. It can even find the concatenated string "stringanother" since that temporary value is also sitting on the stack at the point when we paused our program.

All of these values are called roots. A root is any object that can be reached directly without going through some other object. Most roots are global variables or on the stack, but as we’ll see, there are a couple of other places the VM stores references to objects that it can find. Since the user’s program could possibly read from those variables or stack slots, all roots are considered live and need to be kept around.

This gives us a simple recursive definition of which objects are reachable:

  • All roots are reachable.
  • Any object referred to from a reachable object is itself reachable.

These are the values that are still “live” and need to stay in memory. Any value that doesn’t meet this definition is fair game for the collector to reap. That recursive pair of rules hints at a recursive algorithm we can use to free up unneeded memory:

  1. At a suitable time, run a garbage collection.
  2. Starting with the roots, traverse through object references to find the full set of reachable objects.
  3. Free any objects not in that set.

Later, we’ll get into what a “suitable” time to collect is. This came as a surprise to me, but that turns out to be one of the more difficult, subtle questions to answer when designing a GC.

A number of very different garbage collection algorithms are in use today, but they all roughly follow that same structure. Some may interleave the steps or mix them, but those three fundamental operations are there. They mostly differ in how they perform each step.

Since this book is a guided tour and not a broad survey, we’re going to pick a single algorithm and stick with it. The two most common other algorithms are reference counting (usually shortened to ref counting) and copying collection. If you want a deep dive on those and most of the other strategies, “The Garbage Collection Handbook” is the canonical reference. For a large book on such a deep, narrow topic, it is quite enjoyable to read.

26 . 1 Mark-Sweep Garbage Collection

The first language to not require programmers to explicitly free memory was Lisp, the second “high level” language right after Fortran. John McCarthy considered manual memory management and reference counting, but eventually settled on (and coined) garbage collection—once the program was out of memory it would go back and find unused storage it could reclaim.

In John McCarthy’s “History of Lisp”, he notes: “Once we decided on garbage collection, its actual implementation could be postponed, because only toy examples were being done.” So our procrastination on adding a GC to clox has a luminous legacy.

He designed the very first, simplest garbage collection algorithm, called mark-and-sweep or just mark-sweep. Its description fits in three short paragraphs in the initial paper on Lisp. Despite its age and simplicity, the same fundamental algorithm underlies many modern memory managers. Some corners of CS seem to be timeless.

As the name implies, it works in two phases:

  • Marking. In the first phase, we start with the root values and traverse or trace through all of the value those roots refer to. This is a classic graph traversal of all of the reachable objects. Each time we visit an object, we mark it in some way. (Implementations differ in how they record the mark.)

  • Sweeping. Once the mark phase has completed every reachable object in the heap has been marked. That means any unmarked object is unreachable and ripe for reclamation. We go through them all and free each one.

todo: illustrate phases

A tracing garbage collector is any algorithm that traces through the graph of object references. This is in contrast with reference counting and copying collectors which have different strategies for tracking and visiting the reachable objects.

So that’s what we’re gonna do. Whenever we decide it’s time to reap some bytes, we’ll trace everything and mark all the reachable objects, free what’s didn’t get marked, and then resume the user’s program.

Here, whenever we call reallocate() to acquire more memory, we force a full collection to run. The if check is because reallocate() is also called to free or shrink an allocation. We don’t want to trigger a GC for that—in particular because the GC itself will call reallocate() to free memory.

Checking to begin a collection during allocation is the classic way to wire the GC into the VM. It makes sense for a couple of reasons. You’re already calling into the memory manager to get some bytes, so it’s an easy place to hook in the code. Also, allocation is the only time when you really need to free up some memory, so that you can then reuse it. If you don’t use allocation to trigger a GC, you have to make sure every possible place in code where you can loop and allocate memory also has a way to trigger the collector. Otherwise, the VM can get into a state where it needs more memory but it never reclaims any.

More sophisticated collectors might run on a separate thread, or interleaved periodically during program execution, before function calls, or even when a backwards jump occurs.

26 . 4 Tracing Object References

The next step in the marking process is tracing through the graph of object references from the roots to find every other indirectly reachable object. We don’t have instances with fields yet, so there aren’t many objects that contain references, but we do have some. In particular, ObjClosure has the list of ObjUpvalues it closes over as well as a reference to the raw ObjFunction that it wraps. ObjFunction in turn has a constant table containing references to all of the constants contained in the function’s body. This is enough to build fairly complex webs of objects for our collector to crawl through.

I slotted this chapter into the book right here specifically because now that we have closures, we finally have some objects that contain interesting references to other objects.

Now it’s time to implement that traversal. We can go breadth-first, depth-first, or some other order. Since we just need to find the set of all reachable objects, the order we visit them mostly doesn’t matter. In our GC, we’ll use depth-first because it’s a little simpler to implement a stack than a queue.

I say “mostly” because some garbage collectors copy or move objects in the order they are visited. That means the traversal order determines which objects end up adjacent in memory. That in turn can have a large impact on performance because the CPU uses locality to determine which memory to preload into the caches.

Even when it does matter, it’s not clear which order is best. It’s very difficult to determine which order objects will be used in the future, so it’s hard for the GC to know which order will help performance.

26 . 4 . 1 Tri-color abstraction

Whichever algorithm we choose, the important part is to make sure we don’t lose track of where we are while in the middle of the traversal. It’s less of a concern for our GC since we do the entire marking process in one monolithic step. More advanced implementations like incremental collectors interleave the marking phase with running pieces of the user’s program which means the collector needs to be able to pause and then pick up where it left off later.

To help us soft-brained humans reason about this complex process, GC designers came up with a metaphor called the tri-color abstraction. Each object has a color that helps keep track of what state the object is in and what work is left to do.

Different garbage collection algorithms often add other colors to the abstraction. I’ve seen multiple shades of gray and even purple in some designs. Feel free to choose your own colors if you’re feeling fancy.

  • White – At the beginning of a garbage collection, every object is white. This color means we have not reached or processed the object at all.

  • Gray – During marking, when we first reach an object, it gets darked to gray. This color means we know it is reachable and should not be collected. But we have not yet traced through it to see what other objects it references.

    In graph algorithm terms, this is the worklist. The set of objects we know about but haven’t processed yet. In a depth-first search, this is a stack. In breadth-first, it’s a queue.

  • Black – When we take a gray object and traverse and mark all of the object it references, we then turn it black. This color means the mark phase is completely done with the object and doesn’t need to worry about it anymore.

In terms of that abstraction, the marking process now looks like this:

  1. Start off with all objects white.

  2. Find all the roots and mark them gray.

  3. As long as there are still gray objects:

    1. Pick a gray object. Look at each other object it references. If the referenced object is white, mark it gray.

    2. Mark the original gray object black.

I find it helps to visualize this. You have a web of objects with references between them. Initially, they are all little white dots. Off to the side are some incoming edges from the VM that point to the roots. Those turn gray. Then each gray object’s siblings turn gray while the object itself turns black. The full effect is a gray wavefront that passes through the graph, leaving a field of reachable black objects behind it. Any unreachable object is not touched by this wavefront and stays white.

At the end, you’re left with a sea of reached black objects sprinkled with islands of white objects that can be swept up and freed. Once those are freed, the remaining objects—all black—are reset to white for the next garbage collection cycle.

Most of the other code in here deals with the fact that removing a node from a singly-linked list is kind of cumbersome. We have to continously remember the previous node so we can unlink its next pointer, and we have to handle the edge case where we are freeing the first node. After sweep() completes, the only remaining objects will be the live black ones with their mark bits set. That’s correct, but when the next collection cycle starts, we need every object to be white. So whenever we reach a black object, we go ahead and clear the bit now in anticipation of the next run.

We are almost done. There is one remaining corner of the VM that has some unusual requirements around memory. Recall that when we added strings to clox we made the VM intern them all. That means the VM has a hash table containing a pointer to every single string in the heap. This way when we need a new string, the VM can see if there is already a string with those same characters that it can reuse in order to avoid duplicate strings.

We deliberately did not treat the VM’s string table as a set of roots. If we had, no string would ever be collected. The string table would grow and grow and never yield a single byte of memory back to the operating system. That would be bad.

This can be a real problem. Java does not intern all strings, but it does intern string literals and also provides an API to let users explicitly add strings to the string table. For many years, the capacity of that table was fixed and strings added to it could never be removed. If users weren’t careful about their use of String.intern(), they could inadvertently run out of memory and take down their application.

Ruby had a similar problem for years where symbols were not garbage collected. Both languages (well, their implementations) eventually allowed the GC to collect these strings.

At the same time, if we do let the GC free strings, then the VM’s string table will be left with dangling pointers to freed memory. The next time it looks up a string in the string table, all manner of grievous things could happen. The string table is just special and we need special support for it.

In particular, it needs a special kind of reference. The table should be able to have a reference to a string, but that reference should not be considered a valid edge when determining reachability. That implies that the referred to object can be freed. When that happens, the string table needs to be notified somehow so that it can also clear out the dangling pointer.

This particular set of semantics for a reference comes up frequently enough that it has a name: a weak reference. A weak reference does not keep the referred-to object alive in memory. When the object is freed, the reference itself is removed too. Sort of like a magic self-clearing pointer.

We have already implicitly implemented half of the string table’s unique behavior by virtue of the fact that we don’t traverse it during marking. That means it doesn’t force strings to be reachable. The remaining piece is clearing out any dangling pointers for strings that are freed:

26 . 6 Garbage Collection Strategies

We have a fully functioning mark-sweep garbage collector now. When the stress testing flag is enabled, it gets called all the time, and with the logging enabled too we can watch it do its thing and see that it is indeed collecting. But, when the stress testing flag is off, it never runs at all. It’s time for us to decide when the collector should be invoked during a normal program execution.

As far as I can tell, this question is poorly answered by the literature. When garbage collectors were first invented, computers had a tiny, fixed amount of memory. Many of the early GC papers assume that you set aside a few thousand words of memory—in other words, most of it—and run the collector any time you run out. Pretty simple.

Modern machines have gigs of physical RAM, hidden behind the operating system’s even larger virtual memory abstraction, and is shared among a slew of other programs all fighting for their chunk of memory. The operating system will let your program request as much as it wants and then pages in and out from the disc when physical memory gets full. You never really “run out” of memory, you just get slower and slower.

26 . 6 . 1 Latency and throughput

It no longer makes sense to wait until you “have to”, to run the GC, so we need a more subtle timing strategy. To reason about this more precisely, it’s time to introduce two fundamental numbers used when measuring a memory manager’s performance: throughput and latency.

A managed language, as the name implies, manage’s the program’s heap-allocated memory on the user’s behalf. Users don’t have to explicitly free memory because the VM does it for them. That management comes at a performance cost. The time spent actually freeing memory is the same whether explicitly freed by the user or by the memory manager.

But there is additional overhead the GC adds at runtime to figure out which memory to free. That time spent finding memory to free is time not spent running the user’s code and doing useful work. In our implementation, that’s the entirety of the mark phase. The goal of a sophisticated garbage collector is to minimize that overhead.

There are two key metrics we can use to understand that overhead better:

  • Throughput is the total amount of time spent running user code versus doing garbage collection work. Say you run a clox program for ten seconds and it spends a second of that inside collectGarbage(). That means the throughput is 90%—it spent 90% of the time running the program and 10% on GC overhead.

    Throughput is the most fundamental measure because it tracks the total cost of collection overhead. All else being equal, you want to maximize throughput. Up until this chapter, clox had no GC at all and thus 100% throughput. That’s pretty hard to beat. Of course, it came at the slight expense of potentially running out of memory and crashing if the user’s program ran long enough. You can look at the goal of a GC as fixing that “glitch” while sacrificing as little throughput as possible.

  • Latency is the longest continuous chunk of time where the user’s program was completely paused while garbage collection was happening. It’s a measure of how “chunky” the collector is. Latency is an entirely different metric than throughput. Imagine two runs of a clox program that both take ten seconds. In the first run, the GC kicks in once and spends a solid second in collectGarbage(). It does one big massive collection.

    In the second run, the GC gets invoked five times, each for a fifth of a second. The total amount of time spent collecting is still a second, so the throughput is 90% in both cases. But in the second run, the latency is only 1/5th of a second, five times less than in the first.

If you like analogies—I do—imagine that your program is a bakery selling fresh-baked bread to customers. Throughput is the total number of loaves of warm, crusty baguettes you can serve to customers in a single day. Latency is the how long the unluckiest customer has to wait in line before they get served.

Running the garbage collector is like shutting down the bakery temporarily to go through all of the dishes, sort out the dirty from the clean, and then wash the used ones. In our analogy, we don’t have dedicated dishwashers, so while this is going on, no baking is happening. The bakers are all washing up.

If you consider each person in the analogy to represent a thread, then an obvious optimization is to have separate threads running garbage collection, giving you a concurrent garbage collector. In other words, hire some dedicated dishwashers to wash while others keep baking. Indeed, this is how very sophisticated GCs work, specifically because it does let the bakers—the worker threads—keep running user code almost without interruption.

However, some coordination is required. The dishwashers and bakers do need to put their hands on the same set of dishes. You don’t want a dishwasher grabbing a bowl out of a baker’s hands! This coordination adds overhead and quite a lot of complexity. Concurrent collectors are fast, but challenging to implement correctly.

Reducing the total number of loaves of bread you can sell in a day is bad, and making any particular customer sit and wait while you clean all the dishes is too. The goal is to maximize throughput and minimize latency, but there is no free lunch, even inside a bakery. Garbage collectors make different trade-offs between how much throughput they sacrifice and latency they tolerate.

That’s good because different user programs have different needs. An overnight batch job that is generating a report from a terabyte of data just needs to get as much work done as fast as possible. Throughput is queen. Meanwhile, an app running on a user’s smartphone needs to always respond immediately to user input so that dragging on the screen feels buttery smooth. The app can’t freeze for a few seconds while the GC mucks around in the heap.

Clearly the baking analogy is going to my head.

When the garbage collector runs, it spends some amount of time for each live object it finds, and some amount of time on each dead object that can be reclaimed. In terms of our analogy, maybe you need to spend some time washing the dishes that are dirty and a different amount of time putting setting trays on racks when they still have dough on them.

As a garbage collector author, you control some of the trade-off between throughput and latency by your choice of collection algorithm. A copying collector spends almost no time collecting dead objects but does have to spend time moving every live object it finds in memory. Meanwhile, our mark-sweep collector only needs to briefly visit live objects but has to explicitly call free() on each reclaimed ones.

But even within a single algorithm, we have a lot of control over how frequently the collector runs. Our collector is a “stop-the-world” GC which means the user’s program is paused until the entire garbage collection process has completed. If we wait a long time before we run the collector, then a large number of dead objects will accumulate. That leads to a very long pause while the collector runs and thus high latency.

An incremental garbage collector can do a little collection, then run some user code, then collect a little more, and so on.

So, obviously, we want to run the collector really frequently. But every time the collector runs, it spends some time visiting live objects. That doesn’t really do anything useful (aside from ensuring that they don’t incorrectly get deleted). Time visiting live objects is time not freeing memory and also time not running user code. If you run the GC really frequently, then the user’s program doesn’t have enough time to even generate new garbage for it to collect. The VM will spend all of its time obsessively revisiting the same set of live objects over and over, and throughput suffers.

So, obviously, we want to run the collector really in-frequently. In fact, we want something in the middle, and the frequency of when the collector runs is one of the main knobs we have to tune the trade-off between latency and throughput.

26 . 6 . 2 Self-adjusting heap

We want our GC to run frequently enough to reduce latency but infrequently enough to maintain decent throughput. But how do we find the balance between these when we have no idea how much memory the user’s program needs and how often it allocates? We could pawn the problem onto the user and force them to pick by exposing GC tuning parameters. Many VMs do this. But if we, the GC authors, don’t know how to tune it well, odds are good most users won’t either. We want to have at least a pretty reasonable default behavior.

I’ll be honest with you, this is not my area of expertise. I’ve talked to a number of professional GC hackers—this is something you can build an entire career on—and read a lot of the literature, and all of the answers I got were pretty hand-wavey. The strategy we’re using here is common, pretty simple, and (I hope!) good enough for most uses.

The idea is that the collector frequency automatically adjusts based on the size of the heap. We’ll keep track of the total number of bytes of managed memory that the VM has allocated so far. When it goes above a threshold, we trigger a GC. After that, we note how many bytes of memory remain—how many were not freed. Then we adjust the treshold to some value larger than that.

The result is that as the amount of live memory increases, we collect less frequently in order to avoid sacrificing throughput by re-traversing the live objects. As the amount of live memory goes down, we collect more frequently so that we don’t lose too much latency by waiting too long.

26 . 7 Garbage Collection Bugs

In theory, we are all done now. We have a GC. It kicks in periodically, collects what it can, and leaves the rest. If this were a typical textbook, we would wipe the dust from our hands and bask in the soft glow of the flawless marble edifice we have created.

Alas, I aim to teach you not just the theory of programming languages but the sometimes painful reality. I am going to roll over a rotten log and show you some of the nasty bugs that live under, and garbage collector bugs really are some of the grossest invertebrates out there.

The collector frees any object it can find if it thinks the object is not being used. It is our task to ensure that the garbage collector can find every object it should free. Also, we must take pains to ensure that the GC knows which objects may be used. If we forget to tell it that some object is reachable and the collector frees it, no one is going to be happy when the VM tries to access that later.

In both cases, mistakes are easy to make and not quickly obvious. Slow memory leaks and memory accesses that still look right even though the memory is technically no longer ours can let the VM keep running well past the point that the erroneous code has executed.

This is made even harder by the fact that we don’t know when the collector will run. Any call that eventually allocates some memory is a place in the VM where a collection could happen. It’s like musical chairs. At any point, the GC might stop the music. Every single heap allocated object that we want to use in the future needs to find a chair quickly—get marked as a root or stored as a reference in some other object—before the sweep phase comes to kick it out of the game.

How could the VM use an object that the GC doesn’t see as root? If the GC can’t find it, how can the VM? The most common answer is that a pointer to the object is stored in some local variable on the C stack. The GC walk’s the VM‘s value and CallFrame stacks, but the C stack is hidden to it.

Our GC can’t find addresses in the C stack, but some can. Conservative garbage collectors look all through memory, include the native stack and consider any series of bytes that looks like it could be an address (which is why they are called “conservative”) to be one. The most well-known of this variety is the Boehm–Demers–Weiser garbage collector, usually just called the “Boehm collector”. (The best way to fame in CS is a last name that’s alphabetically early so that it shows up first in sorted lists of names.)

In several previous chapters, we wrote seemingly pointless code that pushed an object onto the VM’s value stack, did a little work, and then popped it right back off. Most times, I said this was for the GC’s benefit. Now you see why. That chunk of work it did between pushing and popping was code that potentially allocated memory and thus could trigger a GC. We had to make sure the object was on the value stack so that the collector’s mark phase would find it and keep it alive.

I wrote the entire clox implementation before splitting it into chapters and writing the prose, so I had plenty of time to find all of these corners and flush out most of these bugs. The stress testing code we put in at the beginning of this chapter and a pretty good test suite were very helpful.

But I only fixed most of them. I left a couple in because I want to give you a hint of what it’s like to encounter these bugs in the wild. If you keep the stress test flag enabled and running some toy Lox programs, you can probably stumble onto a few on your own. Give it a try and then I’ll walk you through them.

That in turn triggers a GC, which fails to mark the new constant object and thus sweeps it right before we have a chance to add it to the table. Crash.

  2. When the sweep phase traverses a live object, it clears the isMarked field to prepare it for the next collection cycle. Can you come up with a more efficient approach?

  3. Mark-sweep is only one of a variety of garbage collection algorithms out there. Explore those by replacing or augmenting the current collector with another one. Good candidates to consider are reference counting, Cheney’s algorithm, or the Lisp 2 mark-compact algorithm.

Design Note: Generational Collectors

A key challenge with how frequently you run the collector is that the “right” answer depends a lot on how much garbage you have and how many objects survive. If most objects are still alive, you don’t want to spend a ton of time revisiting them over and over again. But when a program is churning through temporary objects, you want to collect frequently to keep pauses short and avoid a big backlog of dead bytes.

If only there were some way to tell when or which objects were likely to be long-lived and which weren’t. Well, spoiler alert, there kind of is. Many years ago, GC researchers gathered some metrics on the lifetime of objects in real-world running programs. They tracked every object when it was allocated and eventually when it was no longer needed and then graphed out how long objects tended to live.

They discovered something they called the generational hypothesis, or the much less pleasant term infant mortality. Their observation was that most objects are very short-lived but once they survive beyond a certain age, they tend to stick around quite a long time. This observation is powerful because it gave them a handle on how to partition objects into groups that benefit from frequent collections and those that don’t.

They did this by tracking how old objects are using a system called generational garbage collection. It works like this. Every time a new object is allocated, it goes into a special relatively small region of the heap called the nursery. Since objects tend to die young, the garbage collector is invoked pretty frequently over the objects just in this region.

Also, nurseries are usually managed using a copying collector which is faster at allocating and freeing objects than a mark-sweep collector.

Each time the GC runs over the nursery is called a generation. Any objects that are no longer needed get freed. Those that survive are now considered one generation older, and the GC tracks this for each object. If an object survives a certain number of generations—often just a single collection—it gets tenured. At this point, it is copied out of the nursery into a much larger heap region for long-lived objects. The garbage collector runs over that region too, but much less frequently since odds are good that most of those objects will still be alive.

I love generation collectors because it’s this beautiful marriage of empirical data—the observation that object lifetimes are not evenly distributed—and an algorithm that can take advantage of that fact. It’s also conceptually quite simple. You can think of it as just two separately tuned GCs and a pretty simple policy for moving objects from one to the other.
